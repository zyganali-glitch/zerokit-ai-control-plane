# OpenAI Build Week jury and claim audit

- Audit date: 2026-07-13
- Scope: public `zerokit-ai-control-plane` repository at `main`
- Reviewer lens: senior product engineer + OpenAI Build Week judge proxy
- Important: scores below are an evidence-based reviewer estimate, not an official OpenAI score.

## Official comparison basis

The [OpenAI Build Week page](https://openai.com/build-week/) says submissions are judged on technical implementation, design and user experience, potential impact, and idea quality. It also says strong submissions should use GPT-5.6 and Codex thoughtfully and communicate the problem, solution, and approach clearly.

The [Devpost challenge page](https://openai.devpost.com/) makes the test more concrete:

- **Technological Implementation:** thorough and skillful GPT-5.6 use, genuine effort, working non-trivial code.
- **Design:** a runnable, coherent product experience rather than only a technical proof of concept.
- **Potential Impact:** a credible, specific real problem and audience, supported by what is demonstrated.
- **Quality of the Idea:** a creative, non-obvious GPT-5.6 use and genuine domain understanding.

## Executive verdict

**The repository now supports the narrow claim “a privacy-guarded, GPT-5.6-assisted control-plane configuration workflow with a runnable local preview.” It does not support the broad claim “a production-ready AI control plane that automatically integrates arbitrary SaaS backends.”**

The product idea is competition-relevant and differentiated. The strongest parts are the privacy boundary, route-flexible/payload-strict adapter stance, coherent local preview, and test discipline. The material weakness is evidence, not basic code quality: the repository now contains a real GPT-5.6 Responses API path, but the audit environment's API account returned `429 insufficient_quota`, so no truthful live response manifest could be checked in. A successful funded run is the last P0 proof required before submission.

## Findings fixed in this audit

1. **Static-AI-demo risk:** added an executable GPT-5.6 Responses API runner using the official `gpt-5.6` alias, `store:false`, medium reasoning, and JSON response mode.
2. **No pre-send enforcement:** added a local privacy guard that blocks high-confidence API keys, private keys, tokens, connection strings, cookies, and non-reserved email addresses; identifiers and production-language markers require review.
3. **No generation provenance design:** successful runs now write a secret-free manifest with response ID, requested/returned model, timestamp, storage flag, usage, validation statistics, and SHA-256 hashes—never prompt contents, requirements contents, key, or reasoning.
4. **Static adapter claim:** added a dependency-free PocketBase envelope adapter, synthetic fixture, CLI proof, and fail-closed unit tests.
5. **Demo ambiguity:** added a precise recording roadmap that makes the school scenario primary and PocketBase secondary evidence.

## Remaining submission blockers and priorities

### P0 — before recording or submitting

1. Add API credit/quota to the key used for the demo.
2. Run:

   ```bash
   npm run ai:generate -- ai-buildweek/examples/school-saas.input.md ai-buildweek/evidence/school-saas.gpt-5.6.config.json
   ```

3. Human-review the generated config and manifest; verify no sensitive content and no invented backend compatibility.
4. Re-run all gates and commit both evidence files if valid.
5. Record the generation once without exposing the key or shell environment.

### P1 — materially improves placement

- Publish the static preview to a stable HTTPS URL and put it above the fold in README/Devpost.
- Record a sub-three-minute English demo with captions and a visible end-to-end outcome in the first 30 seconds.
- Add one simple benchmark: manual first-pass estimate versus GPT-assisted generation time, clearly labeled as a single internal exercise rather than a general productivity claim.
- Include the PocketBase adapter proof for 10–15 seconds; do not spend the main narrative configuring PocketBase.

### P2 — post-submission product work

- Add a continuous UI flow for input → privacy preflight → generation → validation → preview.
- Add an executable sanitized OpenAPI comparison engine for more than one backend surface.
- Add eval fixtures for requirement ambiguity, over-permission, unsupported panels, malformed model output, and adversarial privacy inputs.
- Gather customer/developer interviews and measure review time, defect rate, and accepted/rejected AI suggestions.
